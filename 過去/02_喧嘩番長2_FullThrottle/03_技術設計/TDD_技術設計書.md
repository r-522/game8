# 技術設計書 — 喧嘩番長2 Full Throttle

## 対象ハードウェア仕様（PlayStation 2）

| 項目 | 仕様 |
|------|------|
| CPU | MIPS R5900 (Emotion Engine) 294.9MHz |
| メインRAM | 32MB RDRAM (Direct RDRAM) |
| VRAM | 4MB (GS内蔵) |
| グラフィックプロセッサ | Graphics Synthesizer (GS) |
| サウンドプロセッサ | Sound Processing Unit (SPU2) |
| I/O Processor | MIPS R3000A 33.8MHz |
| IOP RAM | 2MB |
| 読み込みメディア | DVD-ROM（片面一層4.7GB / 片面二層8.5GB） |
| 出力解像度 | 480i/480p（標準）/ 最大1080i対応 |
| ポリゴン性能 | 理論値 最大7500万ポリゴン/秒 |
| テクスチャ塗りつぶしレート | 2.4Gpixel/秒 |

### 実質的な制約
- 実ゲームで使用可能なポリゴン数は理論値の10〜15%程度（約750〜1125万ポリゴン/秒が実用上限）
- VRAM 4MBは極めて小さく、テクスチャをタイル分割・ストリーミング管理する必要あり
- メインRAM 32MBからOS・ゲームコード・BGMストリーミングバッファを差し引くと実使用可能量は約20MB前後
- DVD読み込み速度: 最大4.85MB/s（シングルスピード換算4×）

## システムアーキテクチャ概要

```
[ゲームメインループ]
  |-- InputManager        : DualShock2入力ポーリング
  |-- GameStateManager    : 状態機械（FIELD/MENCHI/TANKA/COMBAT/EVENT/MENU）
  |-- FieldSystem         : オープンワールドフィールド管理・NPC配置・時間経過
  |-- MenchiSystem        : メンチビームの判定・タイミングゲーム
  |-- TankaSystem         : タンカ選択肢・正解判定
  |-- CombatSystem        : 3D格闘アクション・ヒット判定・コンボ管理
  |-- OtokogiSystem       : 男気ゲージ計算・イベント監視
  |-- GrowthSystem        : 漢の器・喧嘩魂ポイント管理
  |-- EventSystem         : カットシーン・分岐フラグ管理
  |-- RenderManager       : GS描画コマンド発行・カリング・LOD切替
  |-- AudioManager        : SPU2制御・BGMストリーミング・SEキュー
  |-- SaveSystem          : メモリカード読み書き・フラグシリアライズ
  |-- AssetLoader         : DVD-ROMからのストリーミングロード管理
```

スレッドモデルはPS2の特性上、EEコアが主処理を担い、IOPがDVD読み込みとサウンドストリーミングを非同期処理する二段構成。EE内でもVU0をスキニング計算に、VU1をジオメトリ変換に活用し並列処理を最大化する。

## レンダリングシステム

### ポリゴン数制限

| 対象 | ポリゴン数上限 |
|------|-------------|
| プレイヤーキャラクター | 3,000〜4,000ポリゴン |
| 主要NPCキャラクター | 2,000〜3,000ポリゴン |
| 雑魚NPCキャラクター | 800〜1,200ポリゴン |
| 背景（視錐台内全体） | 15,000〜25,000ポリゴン |
| エフェクト（同時表示） | 2,000ポリゴン上限 |

### テクスチャ解像度
- キャラクターテクスチャ: 256×256 または 512×256（4ビット/8ビットパレット）
- 背景テクスチャ: 256×256（ミップマップ付き）
- UI/HUDテクスチャ: 128×128〜256×256
- フォーマット: PS2独自圧縮（PSM_T4/PSM_T8）

### LOD（Level of Detail）
距離に応じて3段階のLODを切り替える。
- LOD0（近距離〜10m）: フルポリゴン
- LOD1（10〜30m）: ポリゴン数50%削減
- LOD2（30m以上）: ポリゴン数80%削減、テクスチャ解像度1/4

### カリング
- フラスタムカリング（視錐台外オブジェクトを描画リストから除外）
- オクルージョンカリング（建物裏のオブジェクトを除外）
- 距離カリング（50m以上のNPCはスリープ状態に移行）

## 物理演算・衝突判定

### 衝突判定の種類
- **キャラクター同士**: カプセル型コリジョン（高さ1.7m・直径0.5m）
- **フィールド（地面・壁）**: ポリゴンメッシュとAABBの判定
- **攻撃判定（ヒットボックス）**: 攻撃モーション中のみ有効化される球体コリジョン
- **ダメージ判定（ハートボックス）**: キャラクター胴体部の球体コリジョン

### 物理演算方針
PS2の演算リソースを考慮し、本格的なリジッドボディシミュレーションは使用しない。代わりに：
- キャラクター移動はコントローラ入力ベースの直接座標操作
- 吹き飛び・のけぞりはプリセットモーション再生による疑似物理
- 転倒・倒れ込みはラグドールではなく3段階のプリセットアニメーションで対応

## メモリ管理方針

### 全体メモリマップ（推定）

| 領域 | 使用量目安 |
|------|---------|
| ゲームコード本体 | 4〜6MB |
| フィールドジオメトリ（現在エリア分） | 6〜8MB |
| キャラクターデータ（プレイヤー+近距離NPC） | 3〜4MB |
| テクスチャキャッシュ（VRAM補助） | 2〜3MB |
| BGMストリーミングバッファ | 1〜2MB |
| SEキャッシュ | 1MB |
| スタック・ヒープ | 2MB |
| セーブデータワーク | 1MB |
| 予備 | 2MB |
| **合計** | **約22〜27MB** |

### ストリーミング戦略
DVD-ROMから常時非同期でロードを行い、次のエリアデータを先読みする。エリア境界の通路は意図的に細く設計し、先読みバッファに最低2秒以上のロード時間を確保する。

## AI実装方針

### NPC行動AI
有限状態機械（FSM）をベースに、以下の状態を管理する。
- IDLE: 待機（メンチ探知距離外）
- PATROL: ランダムウォーク
- ALERT: プレイヤー発見（メンチ距離内）
- MENCHI: メンチ対決状態
- CHASE: 戦闘開始後の追尾
- ATTACK: 攻撃選択・実行
- STAGGER: のけぞり状態
- RETREAT: HP低下時の後退

### ボスAI
ボスは上記に加え「フェーズ管理」を持つ。HP残量によって攻撃パターンが変化し、プレイヤーの行動を学習して対策を取る（繰り返し使用した攻撃への割込みカウンター等）。

---

## サウンドシステム

SPU2の2つのサウンドコアを活用。コア1をBGMストリーミング（ADPCM形式）に、コア2をSEのポリフォニック再生（同時最大24音）に割り当てる。BGMはDVD-ROMからストリーミング再生し、RAMへの常駐は最小限に抑える。SEはゲーム開始時にIOP RAMへ全量ロードし低レイテンシ再生を保証する。

---

## ロード最適化

- **エリア分割**: オープンワールドを10〜15のサブエリアに分割し、必要エリアのみメモリに保持
- **アーカイブファイル形式**: 独自バイナリパッキング（テクスチャ・ジオメトリ・スクリプト一体化）でDVDシーク回数を削減
- **プリフェッチ**: ストーリー進行フラグに基づき、次の戦闘エリアのデータを事前ロード
- **DVDレイアウト**: 高頻度アクセスデータをDVDの内周側（高速シーク）に配置
