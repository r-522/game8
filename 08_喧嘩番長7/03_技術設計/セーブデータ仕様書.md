# セーブデータ仕様書 — 喧嘩番長7「波嵐烈伝」

## セーブスロット

プレイヤーは3つのセーブスロットを持つ。Supabaseの`kenka_save_slots`テーブルに保存される。

### セーブデータ構造

```typescript
interface SaveSlotData {
  slot_number: 1 | 2 | 3
  player_name: string
  level: number
  exp: number
  hp: number
  max_hp: number
  sp: number
  max_sp: number
  atk: number
  def: number
  spd: number
  ganriki: number
  otokogi: number
  current_district: string
  current_chapter: number
  play_time_seconds: number
  save_count: number
  created_at: string    // ISO8601
  updated_at: string    // ISO8601
}
```

### テーブル定義（kenka_save_slots）
```sql
CREATE TABLE kenka_save_slots (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  player_id uuid REFERENCES kenka_players(id) ON DELETE CASCADE,
  slot_number integer NOT NULL CHECK (slot_number IN (1, 2, 3)),
  player_name text NOT NULL DEFAULT '神威蒼太',
  level integer NOT NULL DEFAULT 1,
  exp integer NOT NULL DEFAULT 0,
  hp integer NOT NULL DEFAULT 100,
  max_hp integer NOT NULL DEFAULT 100,
  sp integer NOT NULL DEFAULT 100,
  max_sp integer NOT NULL DEFAULT 100,
  atk integer NOT NULL DEFAULT 20,
  def integer NOT NULL DEFAULT 10,
  spd integer NOT NULL DEFAULT 10,
  ganriki integer NOT NULL DEFAULT 5,
  otokogi integer NOT NULL DEFAULT 50,
  current_district text NOT NULL DEFAULT 'arasuna',
  current_chapter integer NOT NULL DEFAULT 0,
  play_time_seconds integer NOT NULL DEFAULT 0,
  save_count integer NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(player_id, slot_number)
);
```

## セーブ操作フロー

### 新規セーブ
1. ユーザーがメニュー → セーブタブを開く
2. 空きスロットを選択
3. `savePlayerProgress` Server Actionを呼び出す
4. `kenka_save_slots`にUPSERT（slot_number重複時は上書き）
5. 関連テーブル（skills, items, dachi, story_progress）も同期

### ロード
1. タイトル画面 or メニューからスロットを選択
2. `kenka_save_slots`から該当スロットを取得
3. `gameStore`に全データをセット
4. フィールド画面へ遷移

### オートセーブ
- 章クリア時に自動的に現在のスロットへ上書きセーブ
- バトル中はセーブ不可

## 関連テーブルのセーブ

セーブ時に同期するテーブル一覧：

| テーブル | 説明 | セーブタイミング |
|---------|------|--------------|
| kenka_save_slots | 基本ステータス | 手動 + 章クリア時 |
| kenka_player_skills | 習得スキル一覧 | スキル習得時 |
| kenka_player_items | 所持アイテム | アイテム変動時 |
| kenka_dachi | ダチ一覧 | ダチ変動時 |
| kenka_story_progress | 章進捗 | 章完了時 |
| kenka_player_achievements | 達成実績 | 達成時（即時） |
| kenka_player_legends | 武勇伝達成 | 達成時（即時） |
| kenka_player_subquests | サブクエスト進捗 | クエスト変動時 |
| kenka_battle_records | バトル履歴 | バトル終了時（即時） |
| kenka_rankings | ランキングスコア | 結果画面から（即時） |

## セーブデータの整合性

### 検証ルール
- `level`: 1〜50の整数
- `exp`: 0以上の整数
- `hp`: 1〜max_hp
- `otokogi`: 0〜100の整数
- `current_chapter`: 0〜8の整数
- `slot_number`: 1, 2, 3のみ

### RLS（Row Level Security）
```sql
-- プレイヤーは自分のセーブデータのみ読み書き可能
CREATE POLICY "players can manage own save slots"
  ON kenka_save_slots
  FOR ALL
  USING (auth.uid() = player_id)
  WITH CHECK (auth.uid() = player_id);
```

## バックアップ・復元

### Supabaseバックアップ
- Supabaseの自動バックアップ機能を使用（Pro planで7日間保持）
- 本番環境ではデイリーバックアップを推奨

### データ移行
- ユーザーがデバイスを変えてもログイン（匿名IDを保持）すれば継続可能
- 匿名IDはlocalStorageに保持
- ブラウザキャッシュクリアでIDを失うリスクあり（将来的にメール認証へ移行予定）
