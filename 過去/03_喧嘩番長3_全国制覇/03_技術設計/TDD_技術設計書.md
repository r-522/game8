| 項目 | スペック |
|------|---------|
| CPU | MIPS R4000 @ 333 MHz（通常動作：222 MHz、最大：333 MHz） |
| GPU | カスタムGPU（Curly-Cue VFPU） |

| ストレージ | メモリースティックDuo（セーブデータ保存） |

| 入力 | アナログスティック×1、十字キー、○×△□、L/Rボタン、START/SELECT |
| バッテリー | Li-ion 1800mAh（プレイ時間：約4〜6時間） |

## システムアーキテクチャ概要

```
┌────────────────────────────────────────────────────────────┐
│                     メインゲームループ                         │
│  [GameManager] → [StateManager] → [CurrentGameState]       │
│       ↓                                                     │
│  [InputSystem] → [EventSystem] → [各サブシステム]             │
└────────────────────────────────────────────────────────────┘

[サブシステム一覧]
  - FieldSystem     : フィールド管理・NPC配置・時間進行
  - MenchiSystem    : メンチバトルの入力処理・勝敗判定
  - TankaSystem     : 方言タンカの選択肢管理・正誤判定
  - CombatSystem    : 3D戦闘の入力・ヒット判定・コンボ処理
  - OtokogiSystem   : 男気値管理・行動評価
  - ProgressSystem  : 番長制覇フラグ・しおり収集・エンディング分岐
  - SaveSystem      : セーブ/ロード（メモリースティックI/O）
  - AudioSystem     : BGM・SEの再生・ループ管理
  - RenderSystem    : 3Dレンダリング・スプライト・UIオーバーレイ
```

### ゲーム状態管理（ステートマシン）
- TITLE、MAIN_MENU、FIELD、MENCHI、TANKA、COMBAT、EVENT、INN、RESULT
- 各ステートはenterState / updateState / exitState の3フェーズで管理
- ステート間の遷移はEventSystemを通じて疎結合に設計

## レンダリングシステム

### 3Dキャラクターモデル仕様

| 対象 | ポリゴン数目安 | テクスチャ解像度 |
|------|-------------|---------------|
| 主人公（詳細） | 3,000〜5,000ポリゴン | 256×256（体）+ 128×128（顔） |
| 主人公（戦闘時） | 2,500〜4,000ポリゴン | 同上 |
| 番長（主要） | 2,000〜3,500ポリゴン | 256×256 |
| シャバゾウ（雑魚） | 800〜1,500ポリゴン | 128×128（バリエーション流用） |
| ヒロイン | 2,000〜3,000ポリゴン | 256×256 |

### 背景ステージ仕様

| ステージ | 総ポリゴン数目安 | LOD段階数 |
|---------|--------------|---------|
| フィールド全体（郷都） | 30,000〜50,000ポリゴン | 3段階（近/中/遠） |
| 戦闘アリーナ（1エリア） | 8,000〜15,000ポリゴン | 2段階 |
| 屋内（旅館内部） | 5,000〜8,000ポリゴン | 2段階 |

### レンダリングパイプライン
- ポータルレンダリング（屋内→屋外の描画最適化）
- LOD（Level of Detail）：距離に応じてポリゴン数を3段階切り替え
- テクスチャアトラス：UIテクスチャを1枚にまとめてドローコール削減
- アルファブレンディング：エフェクト・半透明UIに使用（Z-sortが必要なため使用は最小限）

### フレームレート目標
- フィールド探索：30 FPS（固定）
- 戦闘シーン：30 FPS（固定）
- 30 FPS未満になる場合は遠景LODを強制切り替えで対応

## 物理演算・衝突判定

### 衝突判定システム
- キャラクター衝突：カプセルコライダー（高さ×半径）
- 攻撃判定：ボックスコライダー（各攻撃モーションに対応した形状）
- 地形衝突：高さマップ方式（フィールド地形データを事前ベイク）

### 衝突判定の優先順位
1. キャラクター×地形（落下・壁）
2. 攻撃ヒットボックス×被弾ヒットボックス（コンボの高速処理が必要）
3. キャラクター×キャラクター（押し合い）

### 物理演算の範囲
- PSPのCPU性能上、フル物理演算は不可
- 重力・移動はカスタム実装（物理ミドルウェア不使用）
- ラグドール演算なし（倒れるモーションはアニメーション固定）

## メモリ管理方針

### メモリ配分（32MB RAM）

| 用途 | 割り当て量 | 備考 |
|------|----------|------|
| ゲームコード・スタック | 約4 MB | 実行ファイル本体 |
| フィールドデータ | 約6 MB | 現在のマップ1エリア分 |
| キャラクターデータ | 約5 MB | 現在のシーンに出現するキャラ分 |
| アニメーションデータ | 約4 MB | 使用中のモーション分 |
| テクスチャキャッシュ | 約4 MB（VRAM連動） | ページング管理 |
| BGM・SEストリーミングバッファ | 約3 MB | MP3ストリーミング |
| UIデータ | 約2 MB | フォント・UIテクスチャ |
| システム・バッファ | 約4 MB | 予備・OS領域 |

### ストリーミングとページング
- フィールド移動時にロードチョークポイントを設け、次エリアを非同期ロード
- キャラクターモデルはエリア別に必要なものだけを動的にロード/アンロード
- BGMはUMDからストリーミング再生（メモリにすべて展開しない）

## AI実装方針

### シャバゾウ（雑魚敵）AI
- 状態機械：IDLE→PATROL→ALERT→ATTACK→FLEE
- パトロール経路：ウェイポイント方式（固定経路上を往復）
- 攻撃パターン：単純な近距離攻撃のみ。コンボなし
- 多数同時出現時：スロット制（同時に攻撃できるのは2体まで）

### 番長AI（ボス）
- 通常番長：シャバゾウAIに加えて地元スペシャルの発動判断を追加
- 主要ボス（沖田・土方・伊東・近藤）：専用AIスクリプトを実装
  - 体力に応じて行動パターンが変化（体力50%以下で積極攻撃）
  - プレイヤーのコンボを検知してガード/回避を優先する防御モード

### AI処理の最適化
- AI更新フレームをずらす（偶数フレーム組・奇数フレーム組で交互）
- 視野・聴覚判定はシンプルな球体判定（距離チェックのみ）
- ナビゲーションメッシュは事前計算データを使用（リアルタイム経路計算なし）

## サウンドシステム

### 同時発音数
- BGM：1チャンネル（MP3ストリーミング）
- SE：最大8チャンネル同時発音
- ボイス：1チャンネル（AT9形式）

### オーディオデータ形式
- BGM：MP3（128kbps）/ ループポイント設定あり
- SE：VAG（PSP標準音声形式）、8kHz〜44.1kHz
- キャラクターボイス：AT9（ATRAC9、PSP高圧縮形式）

### 3Dポジショナルオーディオ
- 戦闘中のみ2.5D的な距離減衰を適用（PSPスピーカー・ヘッドフォン考慮）
- フィールドBGMは2Dステレオ固定（距離効果なし）

---

## ロード最適化

### ロード時間目標
- 起動〜タイトル：30秒以内
- エリア移動ロード：5秒以内
- 戦闘遷移ロード：3秒以内

### 最適化手法
- UMDの読み取り速度（最大11 MB/s）を考慮したデータ配置
- 頻繁にアクセスするデータ（キャラクターモデル等）をUMDの内周（高速読み取り位置）に配置
- 圧縮：テクスチャはGTF圧縮（PSP対応形式）で容量削減
- セーブデータの書き込みはメモリースティックDuo（バイナリ形式、最大128KB/スロット）
